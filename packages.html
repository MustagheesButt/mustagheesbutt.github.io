SoundCloud is where people from all over the world upload music and sounds they create. Developers can use it in a wide variety of ways, as SoundCloud has an API(Application Programming Interface) which allows developers to get almost all data they want from SoundCloud. Lets take a look at how we can use it!

Knowing the working of an API will be helpful as well as a little knowledge of Asynchronous JavaScript, Promises and callbacks. A little jQuery is used in our code examples, so knowing its basics won’t hurt.

Some beginners get confused between API and SDK(Software Development Kit). Basically, API is what we get data from and SDK is what helps us in doing so. To learn more, checkout the resources at the end of this article.

Getting Started

To start querying the SoundCloud API using JavaScript, we need the JavaScript SDK which is provided by SoundCloud to help developers write code and save time.

I have divided this tutorial in two logical parts. First case, when we do not need user-login feature(allowing user to login to their SoundCloud account). Second, when we need user-login feature.

The Problem

The SoundCloud API documentation uses a different version of SDK(version 3) and the examples use a different and probably older version of SDK(unknown version). Which leads to confusion among developers, especially new ones. Another problem I noticed was that in the latest version of SDK there seems to be an issue with user-login functionality, as the pop-up window, which is supposed to close automatically, does not closes.

The major difference between the two SDKs that I noticed was that the latest version returns a Promise when an asynchronous call to API is made, while the older version requires a callback function as a parameter for an async call.

So for simplicity’s sake, we will use the older version throughout this tutorial, even when user-login functionality is not used.

Using the SoundCloud API

Setup Basic HTML Document

A basic webpage will be required. It can be as simple as below:

<!DOCTYPE html>
	<html>
	<head>
	<title>Include SDK - Using SoundCloud API</title>
	<script src="//connect.soundcloud.com/sdk.js"></script>
	</head>
	<body></body>
	</html>

Notice that we have included SoundCloud SDK in our webpage. You can also download the SDK and reference to it like:

<script src="sdk.js"></script>

To test if the SDK gets loaded in your page, open up the page in a browser. Open up Developer Console. Type SC and press enter. If an undefined kind of error shows up then it is not loading correctly, try refreshing and make sure the path to SDK file(sdk.js) is correct.

Create an App

By “creating an app”, we do not mean to create an Android, Windows Phone or an iOS app. In our context, it refers to getting ourselves, our organization, our website or anything, on whom’s behalf we are going to query an API, registered to the API’s organization(SoundCloud, in our case). It is required because of the authorization protocol, called OAuth(OAuth 2, to be more specific).

Note: We can skip this step, if we are not going to use user-login feature in our website. It will be explained in the next section.

  • We can create our app instantly by heading to this url. Here the apps will be listed which we have already created, if any. Make sure you are logged in to your SoundCloud account.

Note: You do not need to make a separate account for this purpose. You can use the same account which you use for personal purposes.

  • Click on “Register a new application” button.

  • Give it a name and accept SoundCloud’s Developer Policies by checking the checkbox.

  • Click on the big “Register” button, to get registered.

We will be redirected to our newly created app’s settings. We can leave the website and callback fields for now. We’ll get to that later in User login section.

Initialize the Client

By “initializing the client”, we mean to make the client ready to exchange data between itself and SoundCloud servers. We can initialize in our basic HTML document, we created earlier, or in an external .js file, whichever way you like.

The JavaScript syntax to do so is:

SC.initialize({
	client_id: "CLIENT_ID",
	redirect_uri: "CALLBACK_URL"
	});

Lets break it down:

  • SC is an object created by the SDK.
  • The CLIENT_ID is provided to us when we have registered our app.
  • CALLBACK_URL is the url to callback.html, an HTML file which gets called after the user has logged in. We will create it soon.

As was mentioned earlier that in case we are not going to use user-login feature in our site, we do not need to create an app, therefore we will not be provided with a client ID and will not have any callback URL nor a callback file(callback.html) will be required.

In that case we can use anyone’s client ID(but I will recommend creating a separate app and using its client ID) and do not need a callback URL. So in its simplest form, using client ID from SoundCloud’s examples, the initialization code will be something like this:

SC.initialize({
	client_id: "c202b469a633a7a5b15c9e10b5272b78"
	});

Now, after initialization, we are ready to query the SoundCloud API. Move on to the next section to see what we can do already.

Some Demos

If we open browser console and type SC., a list of methods associated with SC object will appear. SC.get(uri, callback) is one of them, which is used for making GET requests to the API.

Getting Some Tracks

To get a list of random tracks, we can use SC.get() like this:

SC.get("/tracks", function(response) {
	  for (var i = 0; i < response.length; i++) {
		$("ul").append("<li>" + response[i].title + "</li>")
	  }
	});

See the Pen List of Tracks - SC API by Mustaghees Butt (@MustagheesButt) on CodePen.

What it does, is that it queries the /tracks endpoint and expects a callback function. The response is stored in the response parameter, which is an array of JavaScript objects having multiple properties, title being one of them. We can console.log(response[0]) instead of the for loop to see a whole object and its properties. Then we can use any number of properties for our purposes.

Playing a Track

SC object offers another method, SC.oEmbed(url, options, callback), which embeds the SoundCloud player in our website and allows us to play a track of our choice.

SC.oEmbed('https://soundcloud.com/username/complete-url-to-the-track', {maxheight: 200, auto_play: false}, function(res) {
	  $("#player").html(res.html);
	});

See the Pen Playing a track - SC API by Mustaghees Butt (@MustagheesButt) on CodePen.

Lets break it up:

  • First we give it a complete URL of the track we want to play.
  • Then we set some options for the player. See more here.
  • Then we replace an element($("#player")) in our webpage with the HTML code for the player(res.html).

User Login

To implement the user login feature, we need to have a callback URL. So if you have not registered/created an app yet, better do it now because the redirect_uri provided during initialization must match the callback url given in our app settings.

Callback

This is the file we need to give in the callback field in our app settings. After a user has logged in, the pop-up window redirects here.

It is just another requirement of OAuth protocol. In our case, we will call it callback.html and it will reside in the same directory as our home page(index.html).

The code for callback file is provided in documentation:

<!DOCTYPE html>
	<html lang="en">
	<head>
	<meta http-equiv="Content-Type" content="text/html; charset=utf-8">
	<title>Connect with SoundCloud</title>
	</head>
	<body onload="window.opener.setTimeout(window.opener.SC.connectCallback, 1)">
	<b style="text-align: center;">This popup should automatically close in a few seconds</b>
	</body>
	</html>

You can modify its message and design, if you would like to.

Logging User in

SC.connect(callback) is the method for implementing user-login feature. It opens up a pop-up window, prompting the user to login to his/her SoundCloud account. Its basic usage is as below:

SC.connect(function () {
	console.log("User has logged in");
	});

A slightly more interesting example would be:

SC.connect(function () {
	SC.get("/me", function (response) {
		console.log("Welcome" + response.username);
	});
	});

After user has logged in, he will be redirected to callback.html we just created, then the pop-up window will automatically close. After that, our callback function will get called. As we can see in the above code, a GET request to /me endpoint is made and as soon as the callback function executes, a welcome message gets logged to the console.

Notice that a request to /me will return data about the currently logged in user. Therefore, using it before the user has even logged in will result in an error message.

Playing with User’s Data

Now, when the user has logged in, we can do much more. To demonstrate some of the power, I have created a demo website on GitHub. You can find its source code here and see it in action here.

Lets walk through it file by file. In index.html, the four divs are of importance, as user data gets filled in them after user has logged in:

<main>

	<div id="ui">
		<h2>Welcome <span></span></h2>
		<img id="avatar" />
		<div id="description"></div>
	</div>

	<!-- TRACKS -->
	<div id="tracklist">
		<h3>Your Tracks:</h3>
		<ul></ul>
	</div>
	<!-- PLAYLISTS -->
	<div id="playlists">
		<h3>Your Playlists:</h3>
		<ul></ul>
	</div>

	<div id="player"></div>

	</main>

The next most important file is script.js.

$(document).ready(function () {
	// Initialization of SDK
	SC.initialize({
		client_id: "21832d295e3463208d2ed0371ae08791",
		redirect_uri: "http://mustagheesbutt.github.io/SC_API/callback.html"
	});

	// Login handler
	var user_perma;
	$("#login").click(function () {
		SC.connect(function () {
			SC.get("/me", function (me) {
				user_perma = me.permalink;
				setUI(me.username, me.avatar_url, me.description);
			});
			if (SC.isConnected) {
				$("header, main").addClass("loggedIn");
			}
			getTracks();
			getPlaylists();
		});
	});

	// set User Interface
	function setUI(userName, avatar, description) {
		$("main h2 span").html(userName);
		$("#avatar").attr("src", avatar);
		$("#description").html(description);
	}
	// function for user tracks
	var tracks = {};
	function getTracks() {
		SC.get("/me/tracks", function (resp) {
			for (var i=0; i< resp.length; i++) {
				$("#tracklist ul").append("<li class='playable'>" + resp[i].title + "</li>");
				tracks[ resp[i].title ] = resp[i].permalink;
			}
		});
	}
	// function for user playlists
	var playlists = {};
	function getPlaylists() {
		SC.get("/me/playlists", function (resp) {
			for (var i=0; i< resp.length; i++) {
				$("#playlists ul").append("<li class='playable'>" + resp[i].title + "</li>");
				playlists[ resp[i].title ] = resp[i].permalink;
			}
		});
	}
	// play something
	function play(uri) {
		url = "http://soundcloud.com/" + user_perma + "/" + uri;
		SC.oEmbed(url, {maxheight: 200}, function (resp) {
			$("#player").html(resp.html);
		});
	}

	// when a track or playlist gets clicked, play it using `play()` function
	$("ul").on("click", function (e) {
		var title = e.target.innerHTML;
		if ( tracks.hasOwnProperty(title) ) {
			play(tracks[title]);
		} else if (playlists.hasOwnProperty(title)) {
			play("sets/" + playlists[title]);
		}
	});

	});

Most of the code is same as it was in case user-login feature is not used. Lets walk through it quickly:

  • First we initialize our app. Notice, this time we have redirect_uri specified to our callback.html.
  • Then we attach a click handler to #login button. Which when clicked, will open a pop-up window prompting user to login.
  • After pop-up window closes, callback function of SC.connect() gets executed.
  • setUI(), getTracks() and getPlaylists(), sets up User Interface, lists user’s tracks and lists user’s playlists respectively.
  • When any track name or playlist name gets clicked, play() function executes, playing that track or playlist using SC.oEmbed() method.

We can get almost all of the user’s data and use it to create a web application sort of thing. For instance, lets get the user’s playlists:

SC.get("/me/playlists", function (response) {
	for (var i =0; i < response.length; i++) {
		console.log(response[i].title);
	}
	});

We can make it part of the User Interface as we like.

We can do much more than that, such as getting or updating user’s description, getting user’s avatar, see who the user is following and his favourites.

Conclusion

  • In case user-login feature is not used, new version of SDK can be used and no need to register an app.
  • Need to register an app, if user-login feature is to be used and use old version of SDK in this case.

Querying an API from the client-side is cool and a powerful tool as it saves us from the complexities of back-end. The SDK makes our lives a whole lot easier. And after we have learned its basics, we can create more powerful and user-friendly web applications. See some at CodePen.
